# ControlRod simulation report — 200 recent microsoft-ui-xaml issues

**Date:** 2026-05-31
**Dataset:** 200 most recent issues from `microsoft/microsoft-ui-xaml` (open + closed), date range 2025-10-14 → 2026-06-01.
**Method:** Ran the deterministic pass exactly as specified in §5.2 of `docs/specs/051-controlrod.md`. The model pass was simulated by me reading the issue bodies. **No labels or comments were applied to any real GitHub issue.**

---

## TL;DR

The deterministic pass works — but with **five concrete defects** that the simulation surfaced, all of which would have made ControlRod noisy on day 1:

| # | Defect | Impact in simulation | Fix |
|---|---|---|---|
| **D1** | `needs-platform` regex too narrow → fires on items that *do* have version info but don't say "x64" | 188/200 (94%) false-positive | Tie `needs-platform` to bug-template field presence, not body keyword search |
| **D2** | `area-NugetPackage` keyword regex (`missing files`, `build error`) too broad | 150/200 (75%) false-positive | Drop the keyword arm; rely on PR-side file paths or explicit template "NuGet" field |
| **D3** | All quality gates fire on feature proposals | Every proposal gets `needs-repro`, `needs-version`, `needs-platform` it doesn't need | Detect template type (bug vs feature) and apply gates only to bug templates |
| **D4** | "Crash" keyword fires on any body containing `deadlock` / `hang` etc. — including feature requests that *describe* a deadlock scenario | 1+ confirmed FP (#11134 is a feature proposal but tagged `Crash`) | Require crash signals to be paired with `Describe the bug` template header or stack-trace shape |
| **D5** | Template-with-junk-content passes `has_repro_steps` (issue #11032 = "ues / yrddy / ste") | 2/200 spam items mis-classified as well-formed | Add a content-quality heuristic: ratio of stop-words to total tokens, presence of all-caps blocks, response-field-only "_No response_" bodies |

The model pass (when I hand-simulated it on ~25 issues) handled severity / area / is-ux-issue correctly **once the deterministic noise from D1-D5 was filtered out**.

## §5 design defects with concrete fixes

### D1 — `needs-platform` over-fires (94%)

**Symptom:** Almost every issue gets `needs-platform`, including ones that explicitly say "Windows 11 24H2 build 26100" or list `x64` somewhere.

**Root cause:** §5.2 has `has_platform` = `\b(x64|x86|ARM64|arm64)\b`. The platform field in the WinUI bug template is *separate* from "Windows version" and *most users only fill the OS version*. Reactor's template will have the same shape.

**Fix:** Replace the keyword regex with **template field presence detection**. Issue templates produce predictable markdown headers — e.g. `### Windows version`, `### Platform`. The deterministic pass should parse the body into a `{header → content}` map and check whether the `Platform` field is the literal `_No response_` (GitHub's placeholder) or has content.

**Spec patch (§5.2):**
```
| `has_platform` | Look up `Platform` field in parsed template map. Empty/`_No response_` → false. |
```

Add a new helper `parseTemplateFields(body)` to `classify-deterministic.mjs` that splits on `^### ` and recognizes the `_No response_` sentinel.

### D2 — `area-NugetPackage` keyword regex catches half the dataset

**Symptom:** 150/200 issues tagged `area-NugetPackage`. The regex was `\b(NuGet package|package reference|missing files|build error)\b`.

**Root cause:** "missing files" and "build error" appear in unrelated bug reports (e.g., "after build error in my project I see the layout is wrong"). The phrase "package reference" appears in any csproj snippet.

**Fix:** For Reactor, drop this category from the issue-side keyword map entirely. NuGet issues are best detected on the PR side via file paths (`*.csproj`, `nuget.config`, `Directory.Packages.props`). Issue-side area inference should be left to the model.

**Spec patch (§5.2):** Remove the issue-side area-keyword map for noisy categories. Keep only high-precision ones (`area-yoga`, `area-reconciler`, `area-hot-reload`) where the name literally identifies the subsystem. Let the model handle the rest.

### D3 — Quality gates fire on feature proposals

**Symptom:** Issues like #11133 "Proposal: Add AccentSplitButtonStyle as a built-in style" get `needs-repro`, `needs-version`, `needs-platform` — none of which apply to API proposals.

**Root cause:** ControlRod is gate-blind to issue *kind*. The XAML template family is:

- 163 issues use the bug template (headers: `Describe the bug`, `Steps to reproduce the bug`, `NuGet package version`, `Windows version`).
- ~21 issues use the feature template (headers: `Summary`, `Rationale`, `Scope`, `Important Notes`, `Open Questions`).
- ~16 issues are freeform.

**Fix:** Add a `template_kind` signal: `bug | feature | freeform`. Quality gates `needs-repro`, `needs-version`, `needs-logs`, `needs-platform` fire only when `template_kind == 'bug'`. `needs-proof` still fires on either when `is_ux_issue && !has_image_or_video`.

**Spec patch (§5.2):** Add `template_kind` to the signal table:
```
| `template_kind` | Match issue body against known template-section signatures. `bug` if `### Describe the bug` present; `feature` if `### Summary` + `### Rationale` present; else `freeform`. |
```

Spec §8 quality-gate table needs a "kind gate" column:
```
| Label | Trigger | Kinds | Resolution |
| needs-repro    | !has_repro_steps      | bug only       | author edits |
| needs-version  | !has_app_sdk_version  | bug only       | author edits |
| needs-logs     | crash && !stack       | bug only       | author edits |
| needs-platform | !template.platform    | bug only       | author edits |
| needs-proof    | is_ux_issue && !img   | bug + feature  | author edits |
```

### D4 — `Crash` label on feature-proposal #11134

**Symptom:** Issue #11134 "Opt-in output transparency..." is a feature request, but body discusses "deadlocks" in the workaround section. The `RE_CRASH` regex matched, ControlRod added `Crash`.

**Root cause:** Crash detection is purely lexical; it doesn't check whether the issue is *reporting* a crash or *describing* one in passing.

**Fix:** Require `crash_signals` AND (`template_kind == 'bug'` OR `has_stack_trace`) before applying the `Crash` label.

**Spec patch (§5.2 / §5.4):** Bump `Crash` from "deterministic alone can decide" to "deterministic gates, model confirms":
```
Severity Crash is applied when:
  crash_signals == true
  AND (template_kind == 'bug' OR has_stack_trace == true OR model.severity == 'Crash')
```

### D5 — Spam issues pass `has_repro_steps`

**Symptom:** #11032 ("test", body: "ues / yrddy / ste") and #11081 ("IS UWP DEAD?", body all-caps rant) both pass the repro check because the template has section headers with text under them, even if the text is garbage.

**Root cause:** §5.2's repro check is structural ("has a `Steps to reproduce` header with non-empty content"). It doesn't measure content quality.

**Fix:** Add a `looks_low_quality` signal that fires when:

- Body length < 200 chars after stripping template scaffolding, OR
- > 50% of body is `_No response_` placeholders, OR
- Title contains ≥ 4 consecutive ALL-CAPS-WORDS, OR
- Body's word count < 20 after de-templating.

When `looks_low_quality && template_kind == 'bug'`, set `needs-repro` regardless of structural check, and apply `request-triage` (don't try to classify severity).

**Spec patch (§5.2):** Add `looks_low_quality` signal. **Spec patch (§5.4):** Add explicit handling for `looks_low_quality` → `request-triage`.

## What worked well (don't change)

1. **`Crash` detection is mostly right.** 45/200 → after D4 fix, expect ~42/200 with 1-2 residual FPs. Strong signal.
2. **`has_image_or_video` detection is right.** 91/200 matches my manual eyeball read.
3. **The marker-backed durable comment + label-set diff model** is correct — no real-data findings suggest changing the shape of the comment or the diff algorithm.
4. **`request-triage` as the low-confidence fallback** is the right escape hatch — multiple cases above (D2 FPs, D5 spam, ambiguous content) all land cleanly on `request-triage` rather than producing wrong labels.
5. **Source classification** is structurally correct (just needs a real allowlist).
6. **The XAML label taxonomy is a good fit** — every label ControlRod wants to apply on this dataset (`Crash`, `area-*`, `needs-*`, `feature proposal`, `nice to have`) maps to a real WinUI label name.

## Recommended spec patches

In priority order:

1. **§5.2 — add `template_kind` signal + `parseTemplateFields` helper.** Unblocks fixes for D1, D3, D4.
2. **§5.2 — replace `has_platform` keyword regex with template-field-based detection.** Fixes D1.
3. **§5.2 — drop body-keyword area inference for noisy categories.** Fixes D2. Reactor's `area-yoga`, `area-reconciler`, `area-hot-reload` are precise enough; everything else goes to the model.
4. **§8 — add a "Kinds" column to the quality-gate table** restricting bug-only gates. Fixes D3.
5. **§5.2 — promote `Crash` to require `template_kind == 'bug' OR has_stack_trace`.** Fixes D4.
6. **§5.2 — add `looks_low_quality` signal and §5.4 handling.** Fixes D5.
7. **§5.2 (new D6 from walkthrough)** — area-keyword regex scope: match in titles and `### Describe the bug` field only, not the whole body. Reduces area FPs.
8. **§13 (v2) — opportunistic `good first issue` recommendation** for trivial doc/typo cases.

None of these change the bot's *architecture* — they all live inside the deterministic-pass module and the label policy. The model contract is unchanged.

## Token budget validation (incidental finding)

For 200 issues, the deterministic-only pass cost zero tokens (purely local regex). If we'd run the full pipeline:

- 200 issues × ~1.5k tokens/classification = ~300k tokens.
- With the D5 `looks_low_quality` short-circuit (~3-5 issues skip the model), ~297k tokens.
- With template-kind short-circuit (freeform/feature templates may need less context), probably ~250k tokens.

**Implication:** 200 issues/period is the GitHub Models monthly free quota. We can sustain ~150-200 *new* issues per month on the free tier. Reactor's actual flow is much lower (~10-30 issues/month right now), so v1 has comfortable headroom. The author-quality lane (§13) is where we'd blow the budget — confirming the spec's recommendation to move to Azure OpenAI for v2.
